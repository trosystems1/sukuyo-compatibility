# architecture.md — sukuyo-compatibility

このファイルは「なぜこの構成にしたか」を記録する。`index.json`が地図（What/Where）なら、こちらは理由（Why）。
コード変更や設計変更の可能性がある作業の前には、エージェントは必ずこのファイルを先に読むこと。

## プロジェクトの目的
竹本光晴堂（kosei-do.co.jp）が提供する宿曜占星術の鑑定サービスのWeb版補助ツール。
本命宿・月のルネーション・対人/対法人の相性診断を、生年月日の入力だけで算出して表示する。

## なぜ単一HTMLファイル構成なのか
- ビルドステップを持たず、`index.html`を直接編集してそのままVercelにpushするだけで反映される運用を優先している。
- フレームワーク（React/Next.js等）を使わず、依存はCDN経由の`lunar-javascript`のみ。
- 理由（推定）：占いツールという性質上、機能追加よりも「文言・スコアの微調整」「UIの見た目調整」の頻度が高く、
  ビルド工程を挟まない方が竹本先生側との細かいやり取りやデプロイサイクルが速い。
- ※この判断の背景（意図的なシンプル化か、単に最初の規模感で始まってそのままなのか）は未確認。次回確認したい。

## 宿曜計算ロジックの設計
- 新暦→旧暦変換は`lunar-javascript`（CDN, v1.7.4）に委譲し、自前実装はしていない。
- 27宿の割り当てロジックは「旧暦の各月1日の宿が固定（1月=室, 2月=奎, …12月=虚）、月内は+1で進む」という規則ベース。
  → `calcShukuIndex`はこの規則をコードに落とし込んだもの。
- 正しさの担保は単体テストではなく、README記載の3つの検証済み確定値（1962/3/22→底宿、1975/8/11→亢宿、1973/1/1→箕宿）による手動確認。
  → ロジックを変更した場合は、必ずこの3パターンで再検証すること。新しいテストケースを思いついたら追加していく方針。

## なぜBasic認証をmiddleware.jsで実装しているか
- 検索エンジンに非公開（vercel.jsonでの非表示設定とあわせて二重に防御）にしたいクローズドな提供物のため。
- Vercelの環境変数（`BASIC_AUTH_USER` / `BASIC_AUTH_PASS`）で本番の認証情報を上書きする運用。
  コード内のデフォルト値（kosei-do / sukuyo2026）はあくまでローカル開発時のフォールバックであり、本番でこのデフォルトのまま運用しないこと。

## 禁止されている操作・注意点
- `SHUKU`配列やひと月の起点となる宿の対応表を変更する際は、必ずREADMEの検証済みテストケースで再検証する。
  （27宿の割り当てがズレると、サービス全体の鑑定結果が全件ズレるため、影響範囲が大きい）
- `package.json`に依存パッケージを追加する場合、ビルドステップが発生しないかを確認すること
  （現状はNo-build運用が前提のため、ビルドが必要になる変更は設計方針との衝突になる）
- middleware.jsのデフォルト認証情報をハードコードのまま本番に出さない。
