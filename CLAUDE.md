# CLAUDE.md — sukuyo-compatibility

## このプロジェクトについて
竹本光晴堂（kosei-do.co.jp）向け、宿曜占星術の本命宿計算・月のルネーション・相性診断ツール。
単一HTMLファイル構成（ビルドステップなし）で、Vercelに直接デプロイする運用。

## Session Start
- シンプルなタスク（文言修正、CSS微調整など）の場合、プロジェクトの全ドキュメントを読む必要はない。
- コード変更の場合、まず関連ファイル、エントリーポイント、マニフェストを検査する（`index.html`内の該当関数、`package.json`）。
- 宿曜計算ロジック、相性スコアの算出方法、UIフロー、デプロイ構成に変更が及ぶ可能性がある場合、
  `index.json`をリポジトリのマップとして使用する。
- 設計判断の理由（なぜ単一ファイル構成か、なぜBasic認証をmiddlewareで実装しているか、
  宿曜計算ロジックの検証方法など）が関わる変更の場合、`architecture.md`を読む。
- コンテキストを目標に絞ること。すべてをロードする代わりに、`grep`や短い行範囲指定を優先する。

## Context7
- 外部ライブラリ、フレームワーク（Vercel Middleware, lunar-javascript等）のAPI・バージョン固有の挙動については
  Context7を使用する。モデルの記憶だけに頼らない。
- プロジェクト固有の決定（宿曜計算ロジック、相性診断の仕様）については、ローカルのこのファイル群を使う。

## 絶対に守ること
- 宿曜計算ロジック（`calcShukuIndex`, `SHUKU`配列, `getShuku`）を変更した場合、
  README記載の検証済みテストケース（1962/3/22→底宿、1975/8/11→亢宿、1973/1/1→箕宿）で必ず再検証する。
- `middleware.js`のBasic認証デフォルト値（kosei-do / sukuyo2026）を本番運用の前提にしない。
  本番はVercel環境変数（`BASIC_AUTH_USER` / `BASIC_AUTH_PASS`）で上書きされている前提で扱う。
- ビルドステップを発生させる変更（npm依存の追加など）は、No-build運用の方針と衝突するため、
  追加前に方針を確認する。
